# 錯誤控制與例外處理 開發規範

目的
- 為了保持專案一致性、可維護性與可靠性，所有開發人員必須遵守本錯誤控制與例外處理規範。

核心原則
1. 所有可預期與不可預期的錯誤點必須以 `Exception`（或語言對應的例外型別）機制處理或傳遞。
2. 不可在邏輯流程中「吞掉」例外或以錯誤控制作為正常流程分支之替代，除非有明確紀錄與設計理由。
3. 例外可在流程中作為非例外分支的工具，但必須保持整體邏輯一致且向同一方向處理（例如：統一在邊界層處理並轉換為使用者可理解的錯誤回應）。
4. 範例程式碼要簡潔明瞭；每個範例只包含一個作法，不要在同一程式碼區段混合多種寫法。

細則與建議
- 捕捉並處理
  - 在可合理恢復或能給出替代行為的位置處理例外。
  - 在無法合理處理時，應記錄必要資訊後重新拋出或包裝成更高階的例外後拋出。
- 不要吞掉例外
  - 不可使用空的 catch 或僅寫註解的 catch，這會掩蓋錯誤並使偵錯困難。
- 記錄（Logging）
  - 例外發生時，應紀錄足夠的上下文（例如：方法名稱、參數摘要、用戶或請求識別、原始例外堆疊）。
  - 日誌等級與內容應視例外嚴重性設定（Error、Warning、Info）。
- 包裝例外（Wrapping）
  - 若要在層間傳遞錯誤，將低階例外包裝成高階業務例外並保留原始例外作為 Inner/InnerException/InnerError，方便追蹤來源。
- 自訂例外型別
  - 當有明確的業務錯誤類別時，建立語意清楚的自訂例外（例如：`ValidationException`、`ResourceNotFoundException`）。
  - 除非必要或經團隊同意，否則避免額外建立自訂例外型別。自訂例外會增加維護成本與範圍複雜度，
    僅在能明確表達業務語意且有跨層處理需求時才建立。
- 例外做為流程控制的限制與允許條件
  - 若在邏輯上利用例外能讓程式碼更簡潔且邏輯更清晰，且該設計經團隊評估接受，允許使用例外作為流程控制的一部分；
    但必須在設計或說明文件中列明原因、適用範圍、替代設計考量與可能的效能影響。
  - 若未達上述條件，例外不應用於常態流程分支（應以一般分支或驗證機制替代）。

快速範例（請注意：每個範例區塊只示範一種做法，簡潔示意）

- 範例 1：捕捉後重新拋出並保留原始例外（C# 範例）
```csharp
public void ProcessFile(string path)
{
    try
    {
        // 可能拋出 IOException
        var content = File.ReadAllText(path);
        // 處理內容...
    }
    catch (IOException ex)
    {
        // 記錄足夠的上下文
        logger.Error(ex, "讀取檔案失敗: {Path}", path);
        // 包裝或重新拋出，保留原始例外
        throw new FileProcessingException("讀取檔案時發生錯誤", ex);
    }
}
```

- 範例 2：不要吞掉例外（示範錯誤做法）
```csharp
try
{
    DoSomething();
}
catch (Exception)
{
    // 錯誤示例：切勿僅註解或空的 catch，會掩蓋錯誤
}
```

- 範例 3：在邊界/入口處統一處理例外並回傳適當回應（示範邊界層處理）
```csharp
// 在 API Controller 或 CLI entry point
try
{
    service.DoWork();
}
catch (ValidationException ex)
{
    // 示範：若系統設計以 ValidationException 表示驗證失敗，
    // 邊界層可將其轉為 BadRequest；此做法需事先團隊同意並於文件說明。
    return BadRequest(ex.Message);
}
catch (Exception ex)
{
    logger.Error(ex, "系統錯誤");
    return InternalServerError("發生未預期錯誤");
}
```

常見反模式
- 使用例外取代常態檢查（例如以 try/catch 代替 if）通常為反模式。
- 只有在能顯著簡化程式且經團隊同意時，才可採用，並且須紀錄理由與效能評估。
- 在深層函式吞掉或忽略例外，使呼叫者無法得知錯誤狀態。
- 過度暴露底層例外給最終使用者（應轉換為友善訊息，同時內部保留詳細日誌）。

檢查清單（開發 PR 時請確認）
- 是否在會失敗或會產生例外的邊界做了適當的捕捉或傳遞？
- 是否避免了吞掉例外？
- 是否有適當的日誌（含上下文）？
- 是否需要建立或使用自訂例外來表達業務語意？（遵循「除非必要或經團隊同意」原則）
- 若以例外作為流程一部分，是否有文件說明理由與影響？

附註
- 規範中所說的「全部包含 Exception 例外處理」是指所有可能拋出例外的邊界與關鍵處理點需有明確處理策略，而不是在每行程式碼都包 try/catch。
