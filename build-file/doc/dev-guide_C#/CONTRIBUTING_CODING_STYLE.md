## CONTRIBUTING_CODING_STYLE.md

## 目的
本文件定義專案的貢獻準則與開發風格規範，確保程式碼一致性、可維護性與可讀性。所有貢獻者（包括外部 PR）應遵守下列規範。

## 命名與風格
- 請遵循專案的 .editorconfig 與 C# 命名慣例。
- 類別、介面、方法命名應具描述性且遵守 PascalCase / camelCase 規則。

## 大括號與縮排
- 大括號（`{` 與 `}`）不可省略。即使區塊只有單一敘述，仍必須使用大括號。
- 大括號各自獨立一行：左大括號獨立成行並置於區塊開始的新行，右大括號獨立成行並與該區塊結束對齊。
- 每個右大括號（`}`）之後，除非下一行緊接為 `else`、`else if`、`catch` 或 `finally`，否則請插入一個空白行。此規則適用於控制區塊、方法、建構子與類別成員之間，且同樣適用於 `}` 與另一個 `}`（例如方法結束後緊接類別結束前亦需插入空白行）。
- `using` 區塊（例如 `using (var r = ...) { ... }`）亦必須使用大括號，並遵循本節關於大括號格式與右大括號後插入空白行的規範。
- `多重using` 區塊必須改成`巢狀單一using`，亦必須使用大括號，並遵循本節關於大括號格式與右大括號後插入空白行的規範。
- - 縮排必須遵循專案的 `.editorconfig` 規則。在建構與格式化流程中應參考並驗證縮排（例如在 CI、pre-commit 或格式化工具中檢查）。
- 若專案尚未定義 `.editorconfig`，預設使用 4 個空格為一級縮排，並在新增 `.editorconfig` 時同步更新本文件。
- `try` / `catch` / `finally` 規則：即使區塊內只有單一敘述，亦不得將整個 `try` / `catch` / `finally` 縮成一行；必須使用大括號且左/右大括號各自獨立成行（例如：

  不允許： `try { DoSomething(); } catch (Exception ex) { Handle(ex); }`

  必須為：

  try
  {
      DoSomething();
  }
  catch (Exception ex)
  {
      Handle(ex);
  }

  當多個 `catch` 或 `finally` 緊接時，可不在 `}` 與下一個 `catch`/`finally` 之間插入空白行，以維持語意緊密；否則請按照控制區塊後的空白行規則插入空白行。）
- `if` / `else if` / `else` 規則：即使區塊內只有單一敘述，亦不得將整個 `if` / `else if` / `else` 縮成一行；必須使用大括號且左/右大括號各自獨立成行（例如：

  不允許： `if(number > 1) number++;`

  必須為：

  if(number > 1)
  {
      number++;
  }
  

  當多個 `else if` 緊接時，可不在 `}` 與下一個 `else if` 之間插入空白行，以維持語意緊密；否則請按照控制區塊後的空白行規則插入空白行。）

## 間距與段落（可讀性提升）
- 類別與函式之間：在不同類別（`class`、`struct`、`interface`）與方法（function / method）之間，應保留至少一行空白以提高段落性與人工閱讀性。
- 類別內成員之間：同一類別內的成員（欄位、屬性、建構子、方法、事件）之間應保留一行空白，以便區隔邏輯區塊。
- 控制區塊後的空白行：在 `if`、`for`、`while`、`switch` 等控制區塊結束（右大括號 `}`）後，除非下一行緊接的是 `else` 或 `else if`，否則必須插入一個空白行以改善段落感與可讀性。對於 `try`/`catch`/`finally`，若下一行緊接的是 `catch` 或 `finally`，可不插入空白行；否則亦應插入空白行。
- 例外情況：為了維持語意的緊密性（例如 `if` 比較緊密地跟着 `else`），可不插入空白行；但對於其它情形應強制插入。
- 自動化檢查：建議在 CI 或 pre-commit 階段使用格式化/分析工具（例如 `dotnet format`、Roslyn 分析器或自訂規則）來驗證上述間距規則，並在 PR 審查中強制執行。
